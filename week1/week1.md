# 1주차-Chapter 1: 개발을 위한 Git과 Github

## Git이란?

> **파일의 변경 사항을 추적**하고 **여러 사용자 간에 해당 파일들의 작업들을 조율**하기 위한 **분산 버전 관리 시스템**
> 

**⇒따라서 여러 사람이 협업하고 버전별 관리가 필요한 프로그래밍 분야에서 강점이 있음.**

파일의 변경사항을 추적: 언제 파일을 수정했는지,  그 때 파일 내에서 어떠한 변화가 있었는지 기록, 또한 **특정 시점에서 이전 내용을 언제든지 다시 꺼내볼 수 있음.⇒**오류관리에 강점: A시점 이후 B시점을 지나 C시점에서 오류가 발견되었다면 변경사항 추적을 통해 오류발견이 용이.

분산버전관리시스템: 여러 명이 한 파일(또는 폴더)에 접근해 개별적으로 작업하고, 작업 결과를 기록으로서 반영하는 시스템

작업 결과를 기록하는 파일→로그(log)라고 부름

## Github

> Git을 이용할 때 쓰 **원격 저장소**
> 

⇒한 파일을 동시에 여러 사용자의 컴퓨터에서 접근하여 완전히 동일한 작업 내용을 공유할 수 있음. 따라서 원활한 협업이 가능

### Github의 작동방식

![image.png](attachment:4b4bf2fd-bdba-4beb-8d94-5dc963c674f3:image.png)

Working directory: 내가 현재 작업하고 있는 폴더

Stage: working directory와 Repository사이에 존재하며 Repository에 본격적으로 commit하기 전 add를 통해 modified된 파일을 선택적으로 업로드할 수 있는 공간

Repository: commit을 통해 staged된 파일들의 변경 사항이 기록되는 공간. push를 통해 원격레포(github에 존재)에 업로드

### .gitignore

워킹 디렉토리 내에서 추적되지 않아야 하는 파일 목록을 지정하는 파일→굳이 추적하지 않아도 될 파일 목록을 지정하고 관리할 수 있음.

## Branch

내가 작업하는 독립적인 영역⇒파일의 구조를 나무로 연상한다면 쉽게 이해가능→branch=나뭇가지 main=나무줄기. 나만 작업하는 branch에서 push를 통해 main에 변경내용을 반영하거나 main의 파일을 clone(복제)시켜 branch에서 독립적으로 작업가능

### git log

커밋해시, 커밋 생성자, 커밋 일시, 커밋 메시지를 확인가능 이때 커밋해시에서 확인할 수 있 **`HEAD`는 현재 브랜치의 최신 커밋를 참조하는 값**으로 이해.

### Issue

팀원들과 프로젝트의 작업 진행 현황과 기능 구현, 버그 수정, 리팩토링 등을 공유하기 위해 생성하는 메모같은 것. 제목, 내용, Assignees, Labels, Projects, Milestone, Development등 프로젝트에 관련된 다양한 내용을 작성할 수 있음.

### PR(pull request)

메인 브랜치에게 Pull을 받아줄 것을 팀원들에 요청(Request)하는 것
