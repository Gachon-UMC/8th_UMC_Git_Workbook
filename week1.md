# 개발을 위한 Git과 Github
## Git이란 무엇이고 왜 사용해야 할까요?
### Git이란?
- 파일 변경 사항을 저장하고 이전 버전들을 체계적으로 관리하는 시스템이다.
### Git을 사용하는 이유
1. 파일의 변경사항을 추적해준다.
    - 즉, 정확히 언제 파일을 수정했는지, 그리고 파일에 어떤 변화가 있었는지 기록해준다.
2. 여러 사용자 간에 해당 파일들의 작업을 조율해준다.
    - Github와 같은 원격 저장소를 여러 사용자가 같이 사용함으로써 여러 사용자가 특정 내용에 대해 접근하고 공동으로 작업할 수 있다.
3. 분산 버전 관리 시스템을 제공한다.
    - 여러 명이 한 파일에 접근해 개별적으로 작업하고, 작업 결과를 버전에 반영한다.

## 파일이 관리되는 방식
### Working Directory
- 로컬 리포지 토리 내에서 내가 현재 작업하고 있는 공간(폴더)을 워킹 디렉토리라고 한다.
### Staging Area
- Working Directory로부터 staging 된 파일이 존재하는 공간이다.
    - staging이란 커밋을 위한 준비 단계로, 파일의 상태를 staged 상태로 바꾸는 작업을 의미한다.
    - add 명령어를 통해 파일을 Staging Area로 옮길 수 있고, 커밋을 가능한 상태로 만든다.
## Local Repository
- 커밋된 파일들이 저장되는 공간이다.
    - 커밋이란, 파일의 변경 사항을 기록하는 것을 의미한다.
Github의 원격 저장소와 LocalRepository를 연결하고, push 명령어를 통해 커밋된 파일은 원격 저장소로 보낸다.
## 브랜치
- 브랜치는 특정 버전 시점에서 서로에게 영향을 끼치지 않도록 작업을 수행하기 위해 사용하는 기능이다.
- 한 브랜치에서의 작업은 기존의 다른 파일들에게 영향을 주지 않는다.
- 따라서 여러 사용자에게 각각 독립적인 작업 환경을 제공한다.
- 독립적인 환경에서 각자 작업을 마친 뒤, 개인의 작업물을 공통 작업에 반영하기 위해 merge를 사용한다.
    - 하지만 여러 사용자가 merge를 사용한다면 충돌 위험이 크기 때문에 보통 merge 명령어 보다는 다른 협업 방식을 사용한다.

## 커밋 기록 확인
- 커밋 기록은 git log를 통해 확인할 수 있다.
    - 언제, 누가, 작성한 커밋 메시지 등에 대한 기록을 모두 볼 수 있다.
    - HEAD를 통해 가장 최신 커밋을 나타내고, 각각의 브랜치가 어느 시점에 위치하는지도 나타낸다.

## Issue와 PR
### Issue
- main 브랜치에서 기능을 나누기 전, 팀원들과 프로젝트 작업 진행 현황, 기능 구현, 버그 수정 등의 정보를 공유할 때 생성한다.
### PR(Pull Request)
- merge는 충돌 위험이 크기 때문에 사전에 메인 브랜치에 나의 커밋을 받아줄 것을 먼저 요청하는 것이 Pull Request이다.
