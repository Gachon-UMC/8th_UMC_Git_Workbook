#Git이란?
: 파일의 변경 사항을 추적하고, 여러 사용자 간에 해당 파일들의 작업들을 조율하기 위한 **분산 버전 관리 시스템**

##분산 버전 관리 시스템이란?
: 쉽게 말하면, 여러 명이 한 파일(또는 폴더)에 접근해 개별적으로 작업하고, 작업 결과를 기록으로 반영하는 시스템

##Working Directory
: 현재 작업하고 있는 공간
* 기본적으로 변경 사항이 모두 추적되지만, 보안 상의 문제로 추적 불가능한 파일들도 존재
	- **.gitignore**파일을 생성해 추적 불가능한(공유 하지 않는)파일들을 넣어줌
	ex) 토큰, 서버의 BaseURL, 또는 충돌 위험이 있어 공유하지 않는 파일

##Staging Area
추적되는 파일들의 상태는 세개로 나뉘는데, 
1. **변경 없음(unmodified)** : 변경되지 않는 파일
2. **변경됨(modified)** : 변경된 파일
3. **스테이지됨(staged)** : 커밋을 하기 위한 준비 상태인 파일
> 커밋(commit)이란?
> : 변경 사항이 생긴**modified**상태일 때 체크포인트를 만드는 일
이때, modified된 파일을 staged 상태로 올리는 작업을 **add**라고 함


##Local Repository
: commit을 통해 파일의 변경 사항을 기록하는 곳

###내 파일 추적 상태 관리하기 
1. unmodified 상태인 파일을 수정하여 modified 상태로 만듦
2. modified된 파일을 add를 통해 staged로 상태로 만듦
	-***git add .***: 모든 파일을 **staged** 상태로 변경하고 싶을 때
	-***git add (파일 이름)***: 특정 파일만 **staged**상태로 변경하고 싶을 때
3. commit을 통해 로컬 레포(local repository)에 반영
	- ***git commit -m "(커밋 메시지)"***
4. push를 통해 원격 레포(remote repository)에 반영
	- ***git push origin (브랜치 이름)***


##브랜치(Branch)
: 개발자들이 동시에 작업하지만 서로에게 영향을 주지 않는 작업을 진행하기 위한 것- 대부분, 기능 단위로 분리
***git branch (브랜치 이름)***: 브랜치 생성
***git checkout (브랜치 이름)***: 브랜치로 이동 & 파일의 수정 내용 복원
***git switch (브랜치 이름)***: 브랜치로 이동
***git restore (브랜치 이름)***: 파일의 수정 내용 복원

***git merge (병합할 브랜치): 다른 브랜치의 변경사항을 현재 브랜치에 적용시키고 싶을 때


##Issue
: 기능 브랜치를 나누기 전 팀원들과 프로젝트의 작업 진행 현황과 긴으 구현, 버그 수정, 리팩토링 등을 공유하기 위한 것
- **제목**: 보통 작업 태그와 대략적인 작업 요약
- **내용**: 컨벤션에 따라 달라짐
- **Assignees**: 이슈를 작업할 담당자 할당
- **Labels**: 작업 태그가 할당
- **Projects, Milestone**: 큰 단위의 작업 내용을 적을 수 있는 부분
- **Development**: 이슈와 연결된 브랜치 or PR 연결

##Pull Request (PR)
: 브랜치를 나누고, 이를 메인 브랜치로 반영하기 위한 작업
- 메인 브랜치에게 Pull을 받아줄 것을 요청하는 것
